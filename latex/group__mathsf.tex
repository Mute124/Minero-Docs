\doxysection{Mathematical Special Functions}
\hypertarget{group__mathsf}{}\label{group__mathsf}\index{Mathematical Special Functions@{Mathematical Special Functions}}
\hypertarget{specfun.h_mathsf_desc}{}\doxysubsection{\texorpdfstring{Mathematical Special Functions}{Mathematical Special Functions}}\label{specfun.h_mathsf_desc}
A collection of advanced mathematical special functions, defined by ISO/\+IEC IS 29124 and then added to ISO C++ 2017.\hypertarget{specfun.h_mathsf_intro}{}\doxysubsubsection{\texorpdfstring{Introduction and History}{Introduction and History}}\label{specfun.h_mathsf_intro}
The first significant library upgrade on the road to C++2011, \href{http://www.open-std.org/JTC1/SC22/WG21/docs/papers/2005/n1836.pdf}{\texttt{ TR1}}, included a set of 23 mathematical functions that significantly extended the standard transcendental functions inherited from C and declared in \texorpdfstring{$<$}{<}cmath\texorpdfstring{$>$}{>}.

Although most components from TR1 were eventually adopted for C++11 these math functions were left behind out of concern for implementability. The math functions were published as a separate international standard \href{http://www.open-std.org/JTC1/SC22/WG21/docs/papers/2010/n3060.pdf}{\texttt{ IS 29124 -\/ Extensions to the C++ Library to Support Mathematical Special Functions}}.

For C++17 these functions were incorporated into the main standard.\hypertarget{specfun.h_mathsf_contents}{}\doxysubsubsection{\texorpdfstring{Contents}{Contents}}\label{specfun.h_mathsf_contents}
The following functions are implemented in namespace {\ttfamily std\+:} 
\begin{DoxyItemize}
\item assoc\+\_\+laguerre -\/ Associated Laguerre functions
\item assoc\+\_\+legendre -\/ Associated Legendre functions
\item beta -\/ Beta functions
\item comp\+\_\+ellint\+\_\+1 -\/ Complete elliptic functions of the first kind
\item comp\+\_\+ellint\+\_\+2 -\/ Complete elliptic functions of the second kind
\item comp\+\_\+ellint\+\_\+3 -\/ Complete elliptic functions of the third kind
\item cyl\+\_\+bessel\+\_\+i -\/ Regular modified cylindrical Bessel functions
\item cyl\+\_\+bessel\+\_\+j -\/ Cylindrical Bessel functions of the first kind
\item cyl\+\_\+bessel\+\_\+k -\/ Irregular modified cylindrical Bessel functions
\item cyl\+\_\+neumann -\/ Cylindrical Neumann functions or Cylindrical Bessel functions of the second kind
\item ellint\+\_\+1 -\/ Incomplete elliptic functions of the first kind
\item ellint\+\_\+2 -\/ Incomplete elliptic functions of the second kind
\item ellint\+\_\+3 -\/ Incomplete elliptic functions of the third kind
\item expint -\/ The exponential integral
\item \doxylink{namespacehermite}{hermite -\/ Hermite polynomials}
\item laguerre -\/ Laguerre functions
\item legendre -\/ Legendre polynomials
\item riemann\+\_\+zeta -\/ The Riemann zeta function
\item sph\+\_\+bessel -\/ Spherical Bessel functions
\item sph\+\_\+legendre -\/ Spherical Legendre functions
\item sph\+\_\+neumann -\/ Spherical Neumann functions
\end{DoxyItemize}

The hypergeometric functions were stricken from the TR29124 and C++17 versions of this math library because of implementation concerns. However, since they were in the TR1 version and since they are popular we kept them as an extension in namespace {\ttfamily \doxylink{namespace____gnu__cxx}{\+\_\+\+\_\+gnu\+\_\+cxx}\+:} 
\begin{DoxyItemize}
\item conf\+\_\+hyperg -\/ Confluent hypergeometric functions
\item hyperg -\/ Hypergeometric functions
\end{DoxyItemize}\hypertarget{specfun.h_mathsf_promotion}{}\doxysubsubsection{\texorpdfstring{Argument Promotion}{Argument Promotion}}\label{specfun.h_mathsf_promotion}
The arguments suppled to the non-\/suffixed functions will be promoted according to the following rules\+:
\begin{DoxyEnumerate}
\item If any argument intended to be floating point is given an integral value That integral value is promoted to double.
\item All floating point arguments are promoted up to the largest floating point precision among them.
\end{DoxyEnumerate}\hypertarget{specfun.h_mathsf_NaN}{}\doxysubsubsection{\texorpdfstring{Na\+N Arguments}{NaN Arguments}}\label{specfun.h_mathsf_NaN}
If any of the floating point arguments supplied to these functions is invalid or NaN (std\+::numeric\+\_\+limits$<$\+Tp$>$\+::quiet\+\_\+\+NaN), the value NaN is returned.\hypertarget{specfun.h_mathsf_impl}{}\doxysubsubsection{\texorpdfstring{Implementation}{Implementation}}\label{specfun.h_mathsf_impl}
We strive to implement the underlying math with type generic algorithms to the greatest extent possible. In practice, the functions are thin wrappers that dispatch to function templates. Type dependence is controlled with std\+::numeric\+\_\+limits and functions thereof.

We don\textquotesingle{}t promote {\ttfamily float} to {\ttfamily double} or {\ttfamily double} to {\ttfamily long double} reflexively. The goal is for {\ttfamily float} functions to operate more quickly, at the cost of {\ttfamily float} accuracy and possibly a smaller domain of validity. Similaryly, {\ttfamily long double} should give you more dynamic range and slightly more pecision than {\ttfamily double} on many systems.\hypertarget{specfun.h_mathsf_testing}{}\doxysubsubsection{\texorpdfstring{Testing}{Testing}}\label{specfun.h_mathsf_testing}
These functions have been tested against equivalent implementations from the \href{http://www.gnu.org/software/gsl}{\texttt{ Gnu Scientific Library, GSL}} and \href{http://www.boost.org/doc/libs/1_60_0/libs/math/doc/html/index.html}{\texttt{ Boost}} and the ratio  \[
  \frac{|f - f_{test}|}{|f_{test}|}
\] is generally found to be within 10\textsuperscript{-\/15} for 64-\/bit double on linux-\/x86\+\_\+64 systems over most of the ranges of validity.
